The CrewAI Framework CrewAI 프레임워크

A code-first runtime for multi-agent systems. Composable primitives (Agent · Task · Crew · Flow), three execution shapes, six integration surfaces. Crew and Flow are independent — either can run on its own. 멀티에이전트 시스템을 위한 코드 우선 런타임. 조합 가능한 기본 단위 (Agent · Task · Crew · Flow), 세 가지 실행 형태, 여섯 가지 연동 표면. Crew 와 Flow 는 독립적 — 각각 단독으로 실행 가능.

02 · Abstractions 추상화 계층

Composable primitives 조합 가능한 기본 단위 — Agent · Task · Crew · Flow

Agent → Task → Crew is the agent execution path; a Crew runs standalone via .kickoff(). Flow is a separate orchestration primitive — it can wrap a Crew, multiple Crews, or just plain Python. The diagram below shows one common composition (Flow wrapping a Crew), but the layers are independent — Crew or Flow can run on its own. Click any rectangle to inspect. Agent → Task → Crew 는 에이전트 실행 경로 — Crew 는 .kickoff() 으로 단독 실행됩니다. Flow 는 별개의 오케스트레이션 기본 단위 — Crew 한 개, 여러 개, 또는 일반 Python 만 감쌀 수도 있습니다. 아래 다이어그램은 가장 흔한 조합 (Flow 가 Crew 를 감싸는 형태) 이지만, 각 계층은 독립적 — Crew 나 Flow 모두 단독 실행 가능. 사각형을 클릭해 확인하세요.

How to read this diagram. Crew and Flow sit side-by-side as peers, not as a hierarchy. Both consume Agent + Task as building blocks. The dashed arrow shows Flow's optional ability to call a Crew — but Flow can run without one. Three valid compositions: (1) Crew alone (run via .kickoff()) · (2) Flow alone (state machine over plain Python) · (3) Flow + one or many Crews. 다이어그램 읽는 법 — Crew 와 Flow 는 위계가 아닌 동등한 peer 로 나란히 배치. 둘 다 Agent + Task 를 기본 단위로 사용. 점선 화살표는 Flow 가 Crew 를 호출할 수 있는 선택적 관계 — Flow 는 Crew 없이도 실행 가능. 유효한 세 가지 조합: (1) Crew 단독 (.kickoff() 으로 실행) · (2) Flow 단독 (일반 Python 위 상태 머신) · (3) Flow + Crew 한 개 또는 여러 개.

03 · Execution Model 실행 모델

Three shapes 세 가지 실행 형태

Same agents and tasks; different orchestration. Sequential and Hierarchical are the two Process types inside a Crew. Flow is a separate primitive — a state machine you can use instead of or alongside a Crew. Switch tabs to compare. 순차 (Sequential) 와 위계 (Hierarchical) 는 Crew 내부의 두 가지 Process 유형. Flow 는 별개의 기본 단위 — Crew 대신 또는 함께 쓸 수 있는 상태 머신입니다. 탭을 전환해 비교하세요.

Sequential 순차 실행

Tasks run in declared order. Each task's output becomes the next task's context. Crew output is the final task's output. 태스크가 선언된 순서대로 실행됩니다. 각 태스크의 output 이 다음 태스크의 context 가 됩니다. 크루의 최종 출력은 마지막 태스크의 출력입니다.
  • Most common shape for pipelines파이프라인에서 가장 자주 쓰이는 형태
  • Easy to reason about, easy to trace추론하기 쉽고 추적하기 쉬움
  • Maps cleanly to Pydantic-typed contractsPydantic 타입 계약과 매끄럽게 연결

Hierarchical 위계 실행

A manager agent receives the task, decides which workers to call (in what order, with what context), and synthesizes the final answer from their outputs. 매니저 에이전트가 태스크를 받고, 어떤 워커를 어떤 순서로 어떤 컨텍스트로 호출할지 결정한 뒤, 결과를 종합하여 최종 답을 만듭니다.
  • Manager picks workers dynamically per request요청마다 매니저가 워커를 동적으로 선택
  • Useful when task decomposition isn't known up front태스크 분해를 사전에 결정할 수 없을 때 유리
  • Workers can themselves be hierarchical (nested)워커 자체가 위계 구조일 수도 있음 (중첩 가능)

Flow 상태 머신

A state machine wrapping Crews and any other Python. @start begins the run, @listen reacts to events, @router picks branches, or_(...) joins paths back together. 크루와 일반 Python 코드를 감싸는 상태 머신입니다. @start 가 실행을 시작하고, @listen 이 이벤트에 반응하며, @router 가 분기를 선택하고, or_(...) 가 경로를 다시 합칩니다.
  • Pre/post-crew Python lives outside the Crew cleanly크루 전후의 Python 로직이 크루 바깥에 깔끔히 위치
  • Conditional branching without hand-rolling control flow제어 흐름을 직접 작성하지 않고 조건 분기
  • Pause / resume via @persist@persist 로 일시정지 및 재개
  • Each @listen step gets its own OTel span각 @listen 단계마다 자체 OTel span 생성

05 · Guardrails 가드레일

4-layer pipeline 4계층 파이프라인 — 한 곳이 아니라 네 곳에서 막는다

Guardrails are not a single chokepoint — four independent layers, each sufficient on its own. If one is bypassed, the next catches it. 가드레일은 한 곳에서 막는 것이 아닙니다. 네 개의 독립적인 계층이 있고 각자가 독립적으로 충분합니다. 한 계층이 뚫려도 다음 계층이 잡습니다.

CrewAI APIs for each layer 각 계층별 CrewAI API

Three CrewAI APIs cover the four layers — a Task(guardrail=...) parameter for task-output checks, and hook decorators from crewai.hooks for tool calls and LLM calls. Each gate auto-retries on False (task-output) or blocks (hooks). 세 가지 CrewAI API 가 네 계층을 모두 커버합니다 — task 출력 검사용 Task(guardrail=...) 파라미터, 그리고 도구 호출 및 LLM 호출용 crewai.hooks 의 훅 데코레이터. task 출력 가드레일은 False 반환 시 자동 재시도되며, 훅은 차단합니다.

Task-output guardrail Task 출력 가드레일
from crewai import Task

def policy_guardrail(output):  # (TaskOutput) → (bool, value_or_error)
    if contains_unmasked_pii(output.raw):
        return (False, "unmasked PII; regenerate")
    return (True, output)

task = Task(
    description="...",
    expected_output="...",
    agent=supervisor,
    guardrail=policy_guardrail,
    guardrail_max_retries=3,  # CrewAI auto-retries
)
Tool-call guardrail 도구 호출 가드레일
from crewai.hooks import before_tool_call, after_tool_call

@before_tool_call
def rbac_check(ctx):  # ctx: ToolCallHookContext
    persona = get_active_persona()  # sourced externally
    if ctx.tool_name not in allowed_for(persona):
        return False  # blocks the call

@after_tool_call
def sanitize(ctx):
    return redact(ctx.tool_result)  # str|None
LLM-call guardrail LLM 호출 가드레일
from crewai.hooks import before_llm_call, after_llm_call

@before_llm_call
def pii_scan(ctx):  # scan ctx.messages before send
    if has_unmasked_pii(ctx.messages):
        return False
    mask_in_place(ctx.messages)

@after_llm_call
def verify_grounded(ctx):  # ctx: LLMCallHookContext
    citations = ctx.agent.knowledge_sources_for(ctx.task)
    if ground_score(ctx.response, citations) > 0.6:
        emit_review(ctx)
    return None  # keep response, just mark
Block reasons logged 차단 사유 기록
Every block records: layer, rule violated, run_id, trace ID, decision timestamp. Hooks emit OTel events automatically. 모든 차단 기록: 계층, 위반 규칙, run_id, trace ID, 결정 시각. 훅은 OTel 이벤트를 자동 발생시킵니다.
Defense in depth 심층 방어
No single layer is the chokepoint. PII spike that slips Layer 1 still gets caught at Layer 4. Tool misuse caught at Layer 2 even if input was clean. 단일 chokepoint 없음. Layer 1 을 통과한 PII 도 Layer 4 에서 잡히고, 입력이 깨끗해도 Layer 2 에서 도구 오용이 차단됩니다.

06 · Korean OCR 한국어 OCR

Field-level confidence 필드 단위 신뢰도 — 라우팅 변수로 사용

Korean OCR itself is solved. The hard part is attaching field-level confidence and using it to route only the uncertain fields to a human — not the whole document. 한국어 OCR 자체는 풀린 문제입니다. 어려운 부분은 필드 단위 신뢰도를 부여하고, 불확실한 필드만 사람에게 보내는 것입니다 — 문서 전체가 아닙니다.

07 · Shared Agent + RBAC 공통 Agent · 다중 사용자 권한

One definition, isolated contexts 정의는 하나, 실행 컨텍스트는 사용자별로 격리

The same agent definition runs under different operator identities. Memory, tool access, and data visibility are all isolated per execution context — without re-instantiating the agent for each user. 동일한 에이전트 정의가 운영자 신원별로 다르게 실행됩니다. 메모리, 도구 접근, 데이터 가시성이 실행 컨텍스트별로 격리되며, 사용자마다 에이전트를 새로 인스턴스화하지 않습니다.

Why this matters 왜 중요한가

Multi-tenant agent deployment without operational cost explosion. Same code, different permissions, isolated audit trails per operator. 운영비 폭증 없이 멀티-테넌트 에이전트 배포가 가능합니다. 코드는 하나, 권한은 사용자별, 감사 추적은 운영자 단위로 격리됩니다.

One definition, N contexts 하나의 정의, N개의 컨텍스트
No per-user agent re-instantiation. Model-call cache + code-maintenance cost do not scale with user count. 사용자별로 에이전트를 새로 만들지 않습니다. 모델 호출 캐시 + 코드 유지비가 사용자 수에 비례하지 않습니다.
Two permission axes 두 축의 권한
Feature (which tool can be invoked?) × Entity (which data rows are visible?). Both axes explicit in the matrix. Feature (어떤 도구를 호출 가능?) × Entity (어떤 데이터 행이 보이는가?). 매트릭스에 두 축 모두 명시.
SCIM auto-sync SCIM 자동 동기화
User/group changes in the IdP propagate to permissions automatically — typical < 5 min. Emergency revoke is instant via token invalidate. IdP 에서의 사용자/그룹 변경이 권한에 자동 반영 — 평균 5분 이내. 긴급 revoke 는 토큰 invalidate 로 즉시.
Audit isolation 감사 격리
Every action carries operator identity + persona + context. Auditor reads themselves go to a separate audit-read log so reads vs writes are distinguishable. 모든 행위에 운영자 신원 + 페르소나 + 컨텍스트가 함께 기록. 감사자의 조회 자체는 별도 audit-read 로그로 — 읽기와 쓰기가 구분 가능.
